## Two layers in one stream

hypernode merges two data sources into a single WebSocket stream:

<CardGroup cols={2}>
  <Card title="Pre-consensus" icon="bolt">
    Cancel and modify actions from the gossip protocol, before validators commit them. ~1 second ahead of the public API.
  </Card>
  <Card title="Confirmed" icon="check">
    Every order, fill, cancellation, and rejection after a block commits.
  </Card>
</CardGroup>

Every event has a `consensus` field: `"pre"` or `"confirmed"`.

## Volume distribution

Based on live mainnet sampling (~6,500 events per 5 seconds):

| Event | Layer | Share | Description |
|-------|-------|-------|-------------|
| `order_rejected` | confirmed | ~35% | Orders rejected (ALO price, margin, etc.) |
| `order` | confirmed | ~16% | New resting limit orders (`data.status: "open"`) |
| `cancelByCloid` | pre | ~11% | Cancel by client order ID |
| `cancel` | pre | ~6% | Cancel by order ID |
| `order_canceled` | confirmed | ~5% | Confirmed cancellation |
| `fill` | confirmed | ~1.5% | Trade execution with PnL |
| `batchModify` | pre | ~1% | Order price/size modification |
| `order_filled` | confirmed | <1% | Immediately filled orders |
| `scheduleCancel` | pre | <1% | Dead man's switch |
| `evmRawTx` | pre | <1% | HyperEVM smart contract calls |

## Confirmed event types

These are produced by the HyperLiquid matching engine after processing each action:

### Orders

| Type | What happened |
|------|--------------|
| `order` | New limit order resting on book (`data.status: "open"`) |
| `order_filled` | Order was filled immediately (IOC, market) |
| `order_canceled` | Order was canceled (confirmed) |
| `order_rejected` | Order was rejected — check `data.status` for reason |
| `triggered` | Trigger/stop order was activated |

Order data fields: `coin`, `side` (A=sell, B=buy), `px` (price), `sz` (size), `oid`, `tif`, `orderType`, `cloid`, `hash`, `builder`, `reduceOnly`, `origSz`.

### Fills

| Type | What happened |
|------|--------------|
| `fill` | A trade was executed |

Fill data fields: `coin`, `side`, `px`, `sz`, `dir` (Open Long/Close Short/etc.), `closedPnl`, `fee`, `feeToken`, `oid`, `tid`, `hash`, `crossed`, `builder`, `builderFee`.

### Rejection reasons

When `type` is `order_rejected`, the `data.status` field contains the specific reason:

| Status | Meaning |
|--------|---------|
| `badAloPxRejected` | ALO order price would cross the spread |
| `perpMarginRejected` | Insufficient margin for perp order |
| `insufficientSpotBalanceRejected` | Insufficient spot balance |
| `iocCancelRejected` | IOC order could not fill |
| `reduceOnlyCanceled` | Reduce-only order has no position to reduce |
| `reduceOnlyRejected` | Reduce-only order rejected |
| `minTradeNtlRejected` | Below minimum trade notional |
| `selfTradeCanceled` | Self-trade prevention triggered |

## Pre-consensus event types

These arrive from the gossip protocol before the validator commits them:

| Type | Description | Data fields |
|------|-------------|-------------|
| `cancel` | Cancel orders by OID | `cancels[].asset`, `cancels[].oid` |
| `cancelByCloid` | Cancel by client order ID | `cancels[].asset`, `cancels[].cloid` |
| `batchModify` | Modify order price/size | `modifies[].oid`, `modifies[].order` |
| `scheduleCancel` | Dead man's switch | Timestamp of scheduled cancel |
| `setReferrer` | Set referral code | Referral code string |
| `evmRawTx` | Pre-block HyperEVM transaction | `data` (raw signed TX hex) |


## Pre-block HyperEVM transactions

Every HyperEVM smart contract interaction appears in the stream as an `evmRawTx` event **before** it is included in a block. HyperLiquid's public RPC does not expose pending transactions (`eth_pendingTransactions` returns "Method not found"), so the hypernode stream is the only way to see these transactions pre-block.

```json
{
  "block": 2641,
  "timestamp": 1776204147470,
  "type": "evmRawTx",
  "consensus": "pre",
  "user": "0x000000360c9f4b00...",
  "data": "f901588206958406194dcd83059491...",
  "network": "mainnet"
}
```

The `data` field contains the full RLP-encoded signed transaction. Decode it with any standard Ethereum library to extract:

- **To address** -- which contract is being called
- **Function selector** -- which function (first 4 bytes of calldata)
- **Calldata** -- full encoded parameters
- **Value** -- ETH/HYPE sent with the call
- **Gas limit and gas price**

### Decoding example (Python)

```python
import rlp
from eth_hash.auto import keccak

def decode_evm_event(event):
    raw = bytes.fromhex(event["data"])
    # EIP-2718 typed transaction (type 2 = EIP-1559)
    tx_type = raw[0]
    decoded = rlp.decode(raw[1:])
    return {
        "tx_hash": "0x" + keccak(raw).hex(),
        "to": "0x" + decoded[5].hex(),
        "value": int.from_bytes(decoded[6], "big"),
        "calldata": decoded[7].hex(),
        "function_selector": "0x" + decoded[7][:4].hex(),
    }
```

### Use cases

- **DeFi monitoring** -- see swaps, lending actions, and liquidity changes on HyperEVM before they settle
- **MEV detection** -- identify arbitrage and liquidation transactions in flight
- **Contract activity tracking** -- monitor specific contracts for pre-block interactions
- **Gas analytics** -- analyze gas pricing patterns on HyperEVM

### Subscribing to EVM transactions only

```json
{"action": "subscribe", "filters": {"action_types": ["evmRawTx"]}}
```

## HIP-4 prediction market events

<Note>HIP-4 prediction markets are currently on **testnet only**. When they launch on mainnet, all prediction market orders and fills will appear in the stream with a `prediction` metadata object.</Note>

Prediction market events use the same types (`order`, `fill`, etc.) but with outcome token coin names (`#39890`) and additional metadata:

```json
{
  "type": "fill",
  "consensus": "confirmed",
  "asset": "#39890",
  "data": {
    "coin": "#39890",
    "prediction": {
      "outcome_id": 3989,
      "side_index": 0,
      "asset_id": 100039890
    }
  }
}
```

See [Prediction Markets](/actions/prediction-markets) for the full reference.
