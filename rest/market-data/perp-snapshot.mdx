The perp snapshot endpoint returns the current clearinghouse state (positions, margin, account value, funding, liquidation price) for **every active trader on HyperLiquid** in a single response. The snapshot is rebuilt in the background every 2-3 seconds and served from a pre-compressed, in-memory cache.

<Note>
Current production snapshot covers **28,828 active traders** (measured live). Refreshed every ~2.5 seconds. Served at ~56ms TTFB from a zero-copy Arc<Bytes> cache. Full payload is 34 MB uncompressed, **~6.4 MB gzipped** (5.4× compression).
</Note>

## Endpoints

| Endpoint | Purpose |
|----------|---------|
| `GET /v2/perp-snapshot/metadata` | Lightweight 110-byte metadata — poll cheaply, fetch full only when fresh |
| `GET /v2/perp-snapshot` | Full snapshot of every active trader's clearinghouse state |
| `GET /v2/spot-snapshot/metadata` | Metadata for spot snapshot |
| `GET /v2/spot-snapshot` | Spot balances for every user with non-empty spot holdings |
| `GET /v2/account-value-snapshot/metadata` | Metadata for account value snapshot |
| `GET /v2/account-value-snapshot` | Projected `accountValue` + `withdrawable` only (small, fast) |

## Metadata-first polling pattern

Polling the full snapshot is expensive (34 MB). Instead, poll the metadata endpoint first — it's 110 bytes and returns instantly. Only fetch the full snapshot when `snapshotTimestamp` changes.

```bash
curl -H "x-api-key: YOUR_KEY" \
  https://hyper.polynode.dev/v2/perp-snapshot/metadata
```

Real response (captured from live production):

```json
{
  "queryMs": 4224,
  "sizeBytes": 34785713,
  "snapshotTimestamp": 1776229734159,
  "totalQueried": 28828,
  "userCount": 28828
}
```

Fields:

<ResponseField name="snapshotTimestamp" type="integer">
  Unix millis when this snapshot was generated. Compare against your last-fetched value — if unchanged, the cached response hasn't been updated, so you can skip fetching the full payload.
</ResponseField>

<ResponseField name="userCount" type="integer">
  Number of traders with non-null clearinghouse state in this snapshot.
</ResponseField>

<ResponseField name="totalQueried" type="integer">
  Number of active addresses we queried (sourced from the live L4 order book).
</ResponseField>

<ResponseField name="queryMs" type="integer">
  Milliseconds it took the background generator to rebuild the snapshot.
</ResponseField>

<ResponseField name="sizeBytes" type="integer">
  Full response size in bytes.
</ResponseField>

## Full perp snapshot

```bash
curl --compressed -H "x-api-key: YOUR_KEY" \
  https://hyper.polynode.dev/v2/perp-snapshot
```

Always pass `--compressed` (or `Accept-Encoding: gzip`) — the server supports gzip and zstd and the payload shrinks from 34 MB to ~6.4 MB on the wire.

## Account value snapshot

If you only need `accountValue` and `withdrawable` per user (for leaderboards, TVL calculations, or quick filters), use `/v2/account-value-snapshot` instead. It's a **12× smaller payload** because it strips positions.

Real response:

```json
{
  "snapshotTimestamp": 1776229734159,
  "userCount": 28828,
  "accounts": {
    "0x000000000005dfe874f01071d0e05f99178a5cfc": {
      "accountValue": "557.607491",
      "withdrawable": "32.235491"
    }
  }
}
```

Full payload is **~2.8 MB uncompressed, ~600 KB gzipped**. Served in ~54ms total.

## Performance

| Endpoint | Uncompressed | Gzipped | TTFB | Total (local) |
|----------|-------------|---------|------|---------------|
| `/perp-snapshot/metadata` | 110 B | 110 B | ~20ms | ~220ms |
| `/perp-snapshot` | 34.8 MB | 6.4 MB | 56ms | 360ms |
| `/account-value-snapshot` | 2.8 MB | 600 KB | ~10ms | 54ms |
| `/spot-snapshot` | 14.3 MB | ~3 MB | ~30ms | ~150ms |

The server handler itself is always <60ms because it just returns a pre-serialized `Arc<Bytes>` from memory. The rest of the total time is network transfer. Always use compression.

## Polling strategy

For active copy trading or whale monitoring:

```python
import asyncio, aiohttp, json

async def watch(session, api_key):
    headers = {"x-api-key": api_key, "Accept-Encoding": "gzip"}
    last_ts = 0

    while True:
        async with session.get(
            "https://hyper.polynode.dev/v2/perp-snapshot/metadata",
            headers=headers
        ) as r:
            meta = await r.json()

        if meta["snapshotTimestamp"] != last_ts:
            async with session.get(
                "https://hyper.polynode.dev/v2/perp-snapshot",
                headers=headers
            ) as r:
                snap = await r.json()
            last_ts = snap["snapshotTimestamp"]
            print(f"Fresh snapshot: {snap['userCount']} users at ts {last_ts}")
            # Process snap["positions"] here

        await asyncio.sleep(1)

async def main():
    async with aiohttp.ClientSession() as session:
        await watch(session, "pn_live_...")

asyncio.run(main())
```

Metadata polls are effectively free (110 bytes), so you can poll at 1 Hz and only fetch the full snapshot when it actually updates (~every 2.5s).

## How it works internally

The snapshot generator runs as a background task inside `hl-l4-api`:

1. **Active address discovery**: every time the L4 order book refreshes (once per second), we extract every unique user address that has a resting order. This gives us a rolling set of currently-active traders.
2. **Batch query**: the generator fires up to 400 concurrent `clearinghouseState` queries against our local HL node (semaphore-limited to prevent overload).
3. **Pre-serialization**: the full response is serialized to JSON once, wrapped in `Arc<Bytes>`, and stored in an `ArcSwap` cell.
4. **Zero-copy serving**: every user request just returns a clone of the `Arc<Bytes>` pointer — no serialization, no I/O, no locking.

The whole cycle takes 2-4 seconds and runs continuously. Snapshot freshness is typically **2-3 seconds**.
